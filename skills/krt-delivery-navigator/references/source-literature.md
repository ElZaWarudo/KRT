# Source Literature

This skill is grounded in software project planning and agile delivery literature. It translates that literature into a compact procedure for turning validated requirements into an execution-ready project plan.

## Primary Basis

- SWEBOK Guide V3, Software Engineering Management (`https://ieeecs-media.computer.org/media/education/swebok/swebok-v3.pdf`): treats software project planning as a core management activity, connects planning to scope, estimation, quality, tools, and review/revision of requirements, and emphasizes that planning depends on visible boundaries and selected processes.
- SEBoK Project Planning (`https://sebokwiki.org/wiki/Planning`): frames planning as an integrated activity covering scope, schedule, resources, risk, quality, verification, validation, and deployment, and stresses that planning evolves with the project.
- The Scrum Guide 2020 (`https://scrumguides.org/docs/scrumguide/v2020/2020-Scrum-Guide-US.pdf`): grounds backlog ordering, sprint planning, product goal alignment, and incremental delivery. It is the main basis for agile backlog and iteration framing in this skill.

## Supporting Basis

- PMI Standards page for the Practice Standard for Work Breakdown Structures (`https://www.pmi.org/standards`): used here as the basis for organizing scope into deliverables and decomposing work without losing the whole-project view.
- PMI WBS discussion articles (`https://www.pmi.org/learning/library/work-breakdown-structure-basic-principles-4883` and `https://www.pmi.org/learning/library/practice-standard-work-breakdown-structures-8063`): reinforce the idea that planning should begin from total scope and deliverables, then decompose to manageable work packages.

## Practical Translation

The skill applies the literature this way:

- **Use requirements as source of truth.** Planning starts from validated requirements rather than re-discovering product behavior.
- **Choose a planning shape intentionally.** Scrum contributes iterative backlog and sprint framing; SEBoK and SWEBOK support broader project planning when a phased or hybrid view is more honest.
- **Plan from deliverables to tasks.** WBS guidance informs the workstream and work-package structure so the plan reflects total scope before detailed activity lists.
- **Make risk, dependencies, and ownership visible.** Planning is not only a schedule; it is a coordinated view of scope, sequence, resources, and uncertainty.
- **Prefer incremental delivery when possible.** Scrum and modern software delivery practice both support using early slices to learn, validate, and reduce risk.
